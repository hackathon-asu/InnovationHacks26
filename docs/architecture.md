## 1. System Overview

Anton RX Track is a **drug coverage tracking and comparison tool**. It ships with a pre-seeded database of real medical policies from major US payers and lets users search drugs, compare coverage across plans, and see what changed between policy versions. Upload and chat are secondary features.

**Architecture**: Dual-stack. A **Python FastAPI backend** handles document ingestion (Docling PDF parsing, NLP pre-extraction, Gemini structured extraction, RxNorm normalization, pgvector embeddings, hybrid RAG). A **Next.js frontend** provides the dashboard, search, comparison matrix, change tracking, and chat UI. Both run on localhost for the demo.

### 1.2 Core User Flows

**Search Drugs (P0)**: User searches by drug name or J-code. Query hits `drug_coverages` table with RxCUI normalization. Results show coverage status across all payers.

**Compare Across Plans (P0)**: User selects a drug. System builds a Drug x Plan comparison matrix showing coverage status, prior auth, step therapy, quantity limits, site-of-care, and approved diagnoses (ICD-10 codes) side by side.

**View Policy Detail (P0)**: User clicks a policy to see extracted drug coverages with hierarchical clinical criteria, step therapy steps, quantity limits, site-of-care restrictions, and provider requirements.

**Track Changes (P0)**: User views a timeline of policy version changes. System shows what changed between versions with structured diffs, significance classification (breaking/material/minor/cosmetic), and LLM-generated summaries.

**Upload Policy (P1)**: User uploads PDF. Docling extracts text with 97.9% table accuracy. scispaCy/Med7 NER pre-extracts drug names and codes. Gemini produces structured JSON grounded in NLP hints. RxNorm normalizes drugs. Results stored in PostgreSQL with pgvector embeddings.

**Ask Questions (P1)**: User asks a natural language question. Hybrid retrieval (structured SQL + pgvector semantic search) finds relevant data. Answer generated by Gemini with citations.

## 4. Data Flow Pipeline

### 4.1 Ingestion Pipeline (8 Stages)

The Python backend runs an 8-stage async pipeline for each policy document. This follows the research document's recommended architecture exactly (§2.3, §9).

```
PDF Document
    |
    v
Stage 1: PARSE (pdf_parser.py)
    |  Docling primary (97.9% table accuracy, preserves Markdown)
    |  → Marker OCR fallback (scanned PDFs, Surya engine)
    |  → pypdf last resort (text-only, no tables)
    |  Section detection via regex (policy_statement, clinical_criteria,
    |    applicable_codes, step_therapy, prior_auth, quantity_limits, etc.)
    |  Returns: ParsedDocument (pages, sections, raw_markdown, metadata)
    |
    v
Stage 2: NLP PRE-EXTRACTION (nlp_extractor.py)
    |  Regex: J-codes (J\d{4}), ICD-10 ([A-TV-Z]\d{2}...), CPT (\d{5})
    |  Optional: scispaCy en_ner_bc5cdr_md (drug + disease NER, F1 ~85.5%)
    |  Optional: Med7 (7 medication entities: drug, dosage, duration, etc.)
    |  Returns: MedNLPResult (drug_names, medical_codes, quantity_hints)
    |
    v
Stage 3: GEMINI STRUCTURED EXTRACTION (gemini_extractor.py)
    |  NLP hints injected into prompt as grounding
    |  Gemini 2.5 Flash, temperature=0.0, max_output_tokens=8192
    |  Long policies split into sections, processed in parallel, merged
    |  Returns: PolicyExtracted with nested:
    |    - drug_coverages (brand, generic, rxcui, j_code, coverage_status)
    |    - clinical_criteria (tree: parent/child, AND/OR/NOT, ICD-10 codes)
    |    - step_therapy (ordered steps with failure definitions)
    |    - quantity_limits (quantity + unit + period)
    |    - site_of_care (preferred, allowed, restricted)
    |    - provider_requirements (specialty, prescriber type)
    |
    v
Stage 4: RXNORM NORMALIZATION (rxnorm_client.py)
    |  NIH RxNav API: drug names → RxCUI, generic names, NDC codes
    |  Rate-limited: 10 concurrent, 3 retries with exponential backoff
    |
    v
Stage 5: SAVE TO POSTGRESQL
    |  Recursive clinical criteria tree insertion
    |  Step therapy, quantity limits, site-of-care, provider requirements
    |  Audit log entry with file hash
    |
    v
Stage 6: CHUNK (chunker.py)
    |  Paragraph-aware sliding window
    |  800 token chunks, 100 token overlap
    |  Preserves page number and section type metadata
    |
    v
Stage 7: EMBED (embedder.py)
    |  Gemini text-embedding-004, 768 dimensions
    |  Batched 50 per request, runs in executor thread pool
    |
    v
Stage 8: INDEX (pgvector)
    |  Stores embeddings in document_chunks table
    |  Metadata filters (payer_name, policy_type, drug_names)
    |  IVFFlat index for cosine similarity search
```

Pipeline progress is tracked per-stage with real-time status polling via API.

### 4.2 Change Detection (change_detector.py)

```
Policy v1 and v2
    |
    v
Section Alignment → Match sections by header + content similarity
    |
    v
difflib SequenceMatcher → Section-level text comparison
    |
    v
Significance Classification:
    - breaking: Drug removed from coverage, new step therapy added
    - material: ICD-10 codes changed, quantity limit changed
    - minor: Wording clarification, date updates
    - cosmetic: Formatting, typos
    |
    v
LLM Summary → Gemini generates human-readable change description
    |
    v
Audit Log → Immutable entry with file hash, diff summary, significance
```

### 4.3 RAG Query Flow (rag_query.py)

```
User Question: "Does Aetna cover Keytruda for breast cancer?"
    |
    v
Query Type Detection (LLM classifies: structured | semantic | hybrid)
    |
    +--> Structured: SQL query on drug_coverages (exact matches)
    +--> Semantic: pgvector similarity search on document_chunks
    |
    v
Result Merge + Reranking
    |  Boost chunks from "clinical_criteria" and "step_therapy" sections
    |
    v
Augmented Prompt → Gemini with structured records + vector chunks as context
    |  temperature=0.1 for factuality
    |  Must include citations (payer, plan, policy number, date)
    |
    v
Streamed Response with Source Citations
```

## 5. Database Schema

PostgreSQL 16 with `pgvector` extension. SQLAlchemy 2.0 async ORM.

### 5.1 Tables (12 total)

| Table | Purpose | Key Fields |
|---|---|---|
| **payers** | Insurance companies | name, plan_types[], metadata |
| **policies** | Policy documents | payer_name, policy_number, title, effective_date, status (pipeline stage), file_hash |
| **policy_versions** | Change history | version_number, diff_json, change_summary, significance |
| **drug_coverages** | Per-drug coverage | brand/generic name, rxcui, j_code, ndc_codes[], coverage_status, prior_auth, raw_extracted |
| **clinical_criteria** | Requirement tree | parent_id (self-ref), criterion_type, logic_operator (AND/OR/NOT), description, is_required |
| **criteria_icd_codes** | ICD-10 per criterion | icd10_code, code_range_start/end |
| **step_therapy** | Ordered steps | step_number, drug_or_class, min_duration, failure_definition, contraindication_bypass |
| **quantity_limits** | Dose/supply limits | quantity, unit, period, max_daily_dose |
| **site_of_care** | Admin restrictions | preferred_site, allowed_sites[], restricted_sites[], exception_criteria |
| **provider_requirements** | Prescriber rules | required_specialty, prescriber_type, consultation_required |
| **document_chunks** | Embeddings for RAG | chunk_text, embedding(768), page_number, section_type, chunk_metadata |
| **audit_log** | Immutable change log | event_type, file_hash, diff_summary, significance |

### 5.2 Key Design Decisions

- **Hierarchical clinical criteria**: Self-referencing `parent_id` enables AND/OR/NOT logic trees. A single drug might have "Must have (Diagnosis A OR Diagnosis B) AND (Failed drug X for 90 days)". This is modeled as a tree, not a flat list.
- **Structured vs JSONB**: Frequently queried fields (coverage_status, prior_auth, rxcui, j_code) are flat columns with indexes. Complex nested data (raw LLM output) stored in JSONB for flexibility.
- **Pipeline status tracking**: The `policies.status` field tracks which pipeline stage a document is in (pending → parsing → nlp_extracting → gemini_extracting → rxnorm → saving_structured → chunking → embedding → indexing → indexed → failed).
- **Audit log with file hash**: Every ingestion creates an audit entry with SHA-256 hash. Designed for future integrity verification.

## 7. LLM Integration

### 7.1 Extraction (Gemini 2.5 Flash, temperature=0.0)

```python
# backend/gemini_extractor.py (simplified)
EXTRACTION_SYSTEM_PROMPT = """
You are a precise medical benefit drug policy parser.
You will receive:
  1. The full text of a medical policy document
  2. NLP pre-extraction hints (drug names, codes found by NER)

Return ONLY valid JSON matching the schema.
Extract ALL drugs. Use NLP hints as grounding — confirm, never ignore.
Prefer exact policy language in description fields.
"""

response = await model.generate_content_async(
    f"{nlp_hints}\n\nPOLICY DOCUMENT:\n\n{policy_text}",
    generation_config=GenerationConfig(temperature=0.0, max_output_tokens=8192),
)
```

**Why temperature=0.0**: Deterministic extraction is critical for change tracking. If the same policy produces different extractions on re-run, version diffs become meaningless noise.

### 7.2 RAG Chat (Gemini 2.5 Flash, temperature=0.1)

Low temperature for factuality. Must cite specific policy numbers, payer names, and effective dates. Hybrid retrieval (SQL + vector) ensures both exact code lookups and semantic understanding.

## 10. ADRs (Architectural Decision Records)

### ADR-001: Dual-Stack Architecture (Python + TypeScript)

**Decision**: Python FastAPI backend for ingestion/RAG, Next.js frontend for UI.

**Why**: Docling (the best PDF parser, 97.9% table accuracy) is Python-only. scispaCy and Med7 are Python-only. Rather than re-implement these in TypeScript with inferior libraries (pdf-parse has no table extraction), we use Python where it's strongest (document processing, NLP) and TypeScript where it's strongest (UI, server components, streaming).

**Trade-off**: Two runtimes to manage locally. Acceptable for localhost demo.

### ADR-002: Docling over pdf-parse

**Decision**: Use Docling (IBM) as primary PDF parser instead of pdf-parse.

**Why**: Medical policy documents have complex multi-column tables (drug formularies, HCPCS code tables, ICD-10 criteria tables). pdf-parse returns flat text with no table structure. Docling uses TableFormer for 97.9% table accuracy and exports structured Markdown that preserves table layout. This is the #1 recommendation from the research document (§1.1).

**Fallback**: Marker (OCR for scanned PDFs) → pypdf (text-only, last resort).

### ADR-003: NLP-Grounded LLM Extraction

**Decision**: Run NLP (scispaCy/Med7/regex) BEFORE Gemini, inject results as grounding hints.

**Why**: Reduces LLM hallucination dramatically. When Gemini sees "NLP PRE-EXTRACTION HINTS: J-codes found: J9271, J0135" it confirms rather than invents codes. Research document recommends this exact pipeline (§2.3). The regex extractors (J-codes, ICD-10, CPT) always run regardless of whether scispaCy is installed.

### ADR-004: Hierarchical Clinical Criteria

**Decision**: Model clinical criteria as a self-referencing tree (parent_id) with AND/OR/NOT logic operators, not a flat JSONB blob.

**Why**: Real medical policies have nested criteria: "Must have (Diagnosis A OR Diagnosis B) AND (Failed Drug X for 90d) AND (Prescribed by specialist)". A flat list loses the logical structure. The tree model enables structured comparison across payers ("Aetna requires 2 step therapy drugs, UHC requires 3") and programmatic evaluation. Research document recommends this (§5.1).

### ADR-005: Free-Tier-Only Stack

**Decision**: All services must have a free tier sufficient for hackathon demo.

**Why**: $0 budget. Gemini 2.5 Flash (500 req/day free), Gemini text-embedding-004 (free), RxNorm API (free, no key), PostgreSQL (Docker, free), Vercel Hobby (free if deployed later). Groq Llama 3.3 70B as backup (6,000 req/day free). Ollama as unlimited local fallback.

### ADR-006: Temperature=0.0 for Extraction

**Decision**: Use `temperature=0.0` for all Gemini structured extraction calls.

**Why**: Deterministic output is critical for change tracking. If the same policy text produces different extractions on re-run, version diffs become meaningless. The change detection system compares structured JSON between versions — any LLM variance creates false positives. Research document notes this concern (§6.2).
